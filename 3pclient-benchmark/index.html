3rd-Party Client Benchmark — Flamegraph Index

Throughput comparison of several S3 clients, with deep-dive profiling into why the CLI tools are slower than the crt-python SDK library. Workload: 10,000 × 1 MiB files (~10 GiB total), upload & download.

Overall Throughput Comparison

All runs: 10,000 × 1 MiB files, concurrency 200, bucket aws-c-s3-test-bucket-269381 (us-west-2). Instance: c7gn.16xlarge (Graviton3, ARM). Note: perf-based profiling below was done on c5n.18xlarge (x86) as perf does not support ARM on this setup — throughput numbers differ between instances.

Client Type Upload Download Upload Download
crt-python Python SDK library 21.8 Gb/s 54.6 Gb/s 🏆
s5cmd CLI tool (Go) 19.7 Gb/s 20.8 Gb/s
rclone CLI tool (Go) 15.4 Gb/s 18.7 Gb/s
cli-crt CLI tool (Python, single-threaded) 4.8 Gb/s 3.7 Gb/s
crt-python is the raw SDK library — it downloads data directly into callbacks with no serialization overhead. The CLI tools (cli-crt, s5cmd, rclone) all do additional work: ListObjects to enumerate files and per-object serialization / deserialization steps. cli-crt is especially slow because that extra work runs on a single Python thread.

Dive Deep: Why Is cli-crt Slow?

cli-crt uses the same underlying CRT library as crt-python but achieves only 3.7 Gb/s download vs 54.6 Gb/s — a 14.8× gap. Two root causes explain the difference:

① Extra serialization steps
The CLI must call ListObjects to discover what to transfer, then serialize/deserialize each S3 request and response through a higher-level API layer. crt-python as a library skips all of this — the caller provides the object list directly and data lands straight in the callback.
② Single Python thread
All that coordination work (listing, scheduling, dispatching) runs on a single Python thread with the GIL. The CRT I/O threads race ahead while Python can't keep up dispatching new requests. Switching to Python free-threading (no GIL, Python 3.14t) partially addresses this but doesn't eliminate the serialization overhead.

How to Read These Flamegraphs

perf flamegraphs were generated from perf record -F 99 --call-graph dwarf and converted to interactive SVG. The x-axis width is proportional to CPU time; the y-axis height is call-stack depth. Colors are arbitrary. Click a frame to zoom; Ctrl+F to search.

py-spy flamegraphs show Python-level call stacks only — native CRT and OS frames are not visible. Use perf to see native overhead; use py-spy to see Python-layer hotspots. Comparing the two reveals exactly how much Python vs C contributes to latency.
